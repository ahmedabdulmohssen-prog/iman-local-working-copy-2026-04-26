# IMAN Current Project Status

Audit date: 2026-05-06

## Executive Status

The current IMAN app is a working local-first financial analysis product with a React/Vite frontend and Express backend. The core loop exists: users enter income and expenses, backend calculations produce deterministic financial outputs, AI is limited to advisor wording, and the frontend displays score, confidence, plausibility guidance, recommendations, investment projections, and manual expense tracking.

The main issue is not missing architecture. The main issue is clarity and trust: planning docs lag behind implementation, score reasoning still needs a more human explanation, and recommendations need more explicit target reasoning.

## What Is Fully Completed

- Core analysis flow is implemented.
  - `artifacts/engineer-money-os/src/App.tsx` posts to `/api/calculate` and `/api/insights`.
  - `artifacts/api-server/src/routes/analyze.ts` exposes both analysis routes.
- Backend calculations are source of truth.
  - `computeTotals` in `artifacts/api-server/src/lib/analysisEngine.ts` owns totals, ratios, opportunities, projections, score, confidence, and plausibility.
- AI is not calculating financial numbers.
  - `/api/insights` computes deterministic opportunities first and asks OpenAI for advisor-note wording only.
- Recommendation intelligence exists.
  - `computeSavingsOpportunities`, `buildShortTerm`, `buildLongTerm`, `fallbackOpportunities`, and the regression suite all show deterministic recommendation selection.
- Recommendation impact math exists.
  - Monthly and yearly impacts are generated by backend helpers and tested.
- Financial score system exists.
  - `computeFinancialScore` returns score, label, interpretation, component breakdown, weakest category, and confidence separation.
- Score explanation UI exists.
  - `ScoreExplanationCard` renders top drivers, component breakdown, and estimated score moves.
- Monthly tracker exists.
  - `EXPENSE_STORAGE_KEY`, `loadStoredExpenses`, `AddExpenseSheet`, and `analyzeWithActualSpending` are implemented in `App.tsx`.
- Live investment projection updates exist.
  - Result view recomputes investment projection locally when `investPct` changes.
- Health routes exist.
  - `artifacts/api-server/src/routes/health.ts` supports `/api/health` and `/api/healthz`.

## What Is Partially Completed

- Recommendation realism is improved but not finished.
  - It now uses deterministic opportunities, impact ranking, confidence-adjusted reductions, and "Why it matters" insight text.
  - It still lacks explicit target reasoning that tells the user why a specific reduction or baseline was chosen.
- Financial score explanation is usable but not yet fully trust-building.
  - It shows score parts, points, and component labels.
  - It does not yet provide a concise plain-English score thesis before the detailed breakdown.
- Monthly tracker is MVP-level.
  - Manual expense logging, local persistence, and actual-spending analysis exist.
  - Weekly summary and on-track/over/under indicators are not implemented.
- Input validation is functional but not polished.
  - Missing income and no-expense states are blocked.
  - Invalid category rows are silently filtered during payload creation.
- Docs are partially current.
  - `PRODUCT_RULES.md` accurately captures the core calculation rules.
  - `TASKLIST.md`, `BACKLOG.md`, `HISTORY.md`, `PROJECT.md`, and `MASTER_CONTEXT.md` contain stale items that now conflict with the code.

## What Is Broken Or Weak

- Stale project-management docs create false priorities.
  - Tracker and live investment projection appear as future work in docs while the code already implements them.
- ZIP code context is not real yet.
  - It is documented as required in `PROJECT.md`, but no ZIP input or cost context engine was found in the source.
- `Credit Score Range` is collected without observed backend effect.
  - This may be intentional future context, but right now it appears to be a user-facing input that does not meaningfully change analysis.
- Financial score weighting is not easy enough to understand.
  - Current backend weights are encoded in functions such as `cashRetentionPoints`, `debtBurdenPoints`, `subscriptionBloatPoints`, `deficitRiskPoints`, and `housingPressurePoints`.
  - The UI exposes parts but not a simple explanation of what the score is rewarding.
- Build/typecheck workflow is fragile in this environment.
  - `pnpm` is not available on PATH.
  - Workspace library type errors prevent a clean project-reference type build.
  - Production build was not run because this task explicitly said not to rebuild the app.

## Technical Debt Risks

- `App.tsx` is very large and owns app shell, forms, tracker, results, and score explanation.
  - Risk: future UI changes can become broad and fragile if not kept tightly scoped.
- Frontend and backend category lists are duplicated.
  - `CATEGORY_NAMES` exists in both `App.tsx` and `analysisEngine.ts`.
  - Risk: category drift would break planned-vs-actual comparisons.
- Planning docs are not reliably treated as source of truth.
  - Risk: future work may reimplement completed systems or prioritize stale tasks.
- Existing workspace library type errors can hide new backend issues.
  - Risk: backend verification may be ambiguous until generated/type outputs are cleaned up.
- `Credit Score Range` may become a trust issue if users expect it to influence recommendations.

## UX Problems

- Score explanation needs a clearer first sentence.
  - Users should understand why they got the score before reading points.
- Recommendation target reasoning is not explicit enough.
  - Users see actions and impacts, but not always the target logic behind the recommendation.
- Input validation feedback is too coarse.
  - Dropped blank or zero rows are not explained inline.
- Tracker status lacks quick interpretation.
  - Actual spending can be analyzed, but the tracker does not yet show simple weekly or category status indicators.
- Planning/document wording still says some built features are missing.
  - This is not user-facing app UX, but it creates execution confusion.

## Architecture Risks

- Adding ZIP code context too early could violate the current rule that backend calculations use only entered data unless the data source and score effect are clearly approved.
- Letting AI expand from wording into action selection would violate `PRODUCT_RULES.md`.
- Adding auth, database sync, bank integration, or OCR now would expand scope before score/recommendation trust is finished.
- Refactoring the app shell before the current clarity work would risk breaking the restored dark UI and working local-first flow.

## Highest ROI Next Steps

1. Fix project docs to match actual implementation.
   - Lowest complexity and prevents duplicate work.
2. Add a concise score thesis to the existing score explanation UI.
   - Use current backend fields; do not change score math.
3. Add deterministic recommendation target reasoning.
   - Use current recommendation engine and expose the reason for each target.
4. Add scenario tests before changing recommendation logic.
   - Protect realism improvements from regressions.
5. Add simple tracker status indicators.
   - Use existing local expenses and planned categories; do not add backend sync.
